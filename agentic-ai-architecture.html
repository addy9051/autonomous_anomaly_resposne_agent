PHASE 00

Research & Planning

Domain analysis, stakeholder mapping, problem scoping, risk assessment, technology evaluation, and success metrics definition before any code is written.

Domain & Pain Point Analysis

Core Pain Points
  • Sub-second SLA breach: Payment approvals must complete in <1s — any latency spike causes cascading failures and customer churn
  • 24/7 alert fatigue: Human on-call engineers face thousands of low-signal alerts; mean-time-to-acknowledge (MTTA) is 12–20 min
  • Fraud signal drift: Fraud patterns evolve faster than rule-based systems are retrained — stale models miss emerging attack vectors
  • Data quality degradation: Upstream schema changes, null injections, and volume spikes silently corrupt downstream analytics
  • Manual runbook execution: Diagnosis and remediation steps are documented but require human interpretation and action
Stakeholder Map
  • SRE/Platform Team: Primary consumers — need accurate triage, automated remediations, and reduced pager fatigue
  • Payments Engineering: Owns transaction gateway — needs anomaly attribution and schema change awareness
  • Fraud Risk Team: Requires real-time fraud signal enrichment and adaptive model updates
  • Customer Operations: Benefits from automated CS queue deflection when incidents are detected early
  • Security & Compliance: Must audit all autonomous actions; requires explainability and full audit trails
  • Executive/Finance: Cares about uptime SLAs, cost of incidents, and MTTR reduction ROI

Success Metrics & SLOs

MTTA

Mean-time-to-acknowledge reduced from 15 min → <90 seconds via autonomous triage

MTTR

Mean-time-to-remediate reduced from 45 min → <8 min for known-pattern incidents

False Positive Rate

<5% false-positive triage actions; validated via confusion matrix on historical incidents

Data Quality Score

Anomaly detection recall ≥ 95% on transaction schema violations; latency <200ms per record

Risk Register

Autonomous Action Risk

Agent triggers incorrect scale-up or restarts production services. Mitigation: mandatory human-in-the-loop gate for destructive actions; dry-run mode with shadow execution.

LLM Hallucination

Diagnosis agent fabricates runbook steps not grounded in the vector store. Mitigation: RAG grounding validation, confidence thresholds, and structured output schemas (Pydantic).

Data Drift in RL Loop

Feedback loop reinforces suboptimal actions if reward signal is noisy. Mitigation: reward shaping with SRE oversight; A/B test new policies before full rollout.

Data Extraction Strategy

Real-time Extraction (Streaming)
  • Kafka Connect with JDBC source connectors for transactional DBs; change-data-capture (CDC) via Debezium for PostgreSQL WAL streaming
  • OpenTelemetry SDK auto-instrumented on all payment services; spans exported to OTLP collector sidecar
  • Prometheus remote_write to Thanos Receive for long-term metric retention; recording rules pre-aggregate high-cardinality series
  • Webhook receivers for PagerDuty, Slack, and CS platforms — event normalization via a schema-registry-backed Avro serializer
Batch Extraction (Knowledge Base)
  • Confluence / Notion API polled nightly; changed pages re-chunked (512 tokens, 64 overlap) and re-embedded via text-embedding-004
  • PagerDuty Incidents API backfills 2 years of incident history for RL reward model training
  • JIRA / ServiceNow ticket corpus extracted for CS pattern clustering — used to pre-warm diagnosis agent context
  • GitHub Actions / Jenkins deployment event feed; correlated with incident timelines for change-failure-rate analysis

Cross-Cutting Concerns

Security & Access Control
  • All agents run with least-privilege service accounts (GCP IAM / AWS IAM roles)
  • Secrets managed via HashiCorp Vault; rotated every 24h
  • Agent-to-agent comms over mTLS; API gateway with JWT validation
  • PII masking in all LLM prompts — txn amounts, card numbers, customer IDs tokenised before sending to LLM APIs
Observability Stack
  • OpenTelemetry for end-to-end agent trace propagation (trace every LLM call with span metadata)
  • Grafana dashboards: agent decision latency, tool call success rates, false-positive rates
  • LLM cost tracking via LangSmith / Helicone; per-agent token spend alerts
  • Slack/PagerDuty escalation when agent confidence <70% on any triage decision
State Management
  • LangGraph checkpointing to Redis for in-flight agent state; Postgres for durable incident state
  • Agent memory: episodic (current incident context in Redis), semantic (runbook embeddings in pgvector), procedural (RL policy weights in Vertex AI)
  • Conversation context window managed via sliding window + summarisation; max 32K tokens per incident thread

Infrastructure Blueprint

GCP-Primary Deployment
  • GKE Autopilot: Agent pods with resource requests (2 vCPU, 4GB) and HPA on queue depth
  • Cloud Run: Lightweight tool-execution lambdas (sub-agent API calls)
  • Vertex AI: LLM hosting (Gemini 1.5 Pro/Flash), Matching Engine for vector search, RL jobs
  • Cloud Spanner: Global incident state store (multi-region, 99.999% SLA)
  • Pub/Sub + Dataflow: Agent event bus; Dataflow for stream processing at scale
Multi-Cloud & Hybrid
  • AWS Bedrock: Titan embeddings as backup; Bedrock Agents for AWS-native resource remediation (EC2, RDS, ECS scaling)
  • Azure AI Services: Azure OpenAI fallback; Azure Personalizer for feedback loop variant; Teams webhook integration
  • On-prem Kafka cluster: PCI-compliant network zone; no raw payment data leaves on-prem; anonymised events forwarded to cloud
  • N8n self-hosted: All automation workflows run on-prem to access internal APIs without VPN overhead

Agent Internal Design

🔭
Monitoring Agent
langchain · autogen · prometheus-tools

Trigger: Kafka consumer on telemetry.normalized topic. Processes each event through a LangChain tool-calling loop with a ReAct reasoning chain.

# Core tool definitions
tools = [
  PrometheusQueryTool(endpoint=PROM_URL),
  KafkaLagInspectorTool(brokers=BROKERS),
  FraudSignalFetchTool(store=REDIS),
  BaselineCompareTool(window="7d"),
  AnomalyClassifierTool(model="iso_forest_v3")
]
# Confidence gate: escalate if score > 0.75
  • Anomaly Detection: Isolation Forest + LSTM autoencoder for time-series; statistical z-score for simple threshold breaches
  • Natural Language Reasoning: ReAct prompt — agent explains why it flagged the anomaly before escalating
  • Output Schema: AnomalyEvent(id, severity, affected_service, metrics_snapshot, reasoning, confidence)
🔬
Diagnosis Agent
langgraph · crewai · rag · sub-agents

Architecture: LangGraph DAG with 4 nodes: context_gather → rag_lookup → sub_agent_dispatch → synthesis. CrewAI coordinates 3 specialised sub-agents.

class DiagnosisGraph(StateGraph):
  nodes = [
    "gather_context", # pull metrics window
    "rag_runbook_lookup", # pgvector similarity
    "dispatch_subagents", # CrewAI crew
    "synthesise_rca" # structured output
  ]
  # State persisted in Redis checkpoint
  • Network Sub-Agent: Checks DNS resolution times, CDN edge latency, BGP route anomalies, firewall rule changes
  • Database Sub-Agent: Queries slow query log, lock wait graphs, connection pool saturation, replication lag
  • Application Sub-Agent: Inspects pod crash logs, OOM events, deployment diffs from last 30 min, circuit breaker states
  • RAG Config: Top-5 runbook chunks by cosine similarity; reranked by BM25 keyword overlap with anomaly description
⚙️
Action Agent
n8n · incident-api · slack · autogen

Decision Gate: Actions classified into 3 tiers by blast radius. Tier 1 executes autonomously; Tier 2 requires async SRE approval via Slack interactive message; Tier 3 always pages on-call.

ACTION_TIERS = {
  "tier1_auto": ["scale_replicas", "clear_cache",
                 "restart_unhealthy_pod", "update_circuit_breaker"],
  "tier2_approve": ["drain_node", "failover_db",
                   "throttle_merchant"],
  "tier3_human": ["rollback_deployment", "block_issuer",
                 "network_policy_change"]
}
  • N8n Workflows: 14 pre-built automation workflows triggered via webhook; each workflow idempotent with rollback capability
  • Incident Summary: Autogen drafts plain-language incident summary (≤200 words) with RCA, actions taken, and next steps
  • Output: Incident record written to Spanner; Slack thread updated with status and timeline
🔄
Feedback Loop Agent
vertex ai rl · reward shaping · policy update

Architecture: Contextual bandit model on Vertex AI. Observes (state → action → outcome) triplets; reward signal shaped by MTTR reduction and false-positive rate.

# Reward function
def compute_reward(incident):
  r = 0.0
  if incident.auto_resolved:
    r += 1.0 - (incident.ttm / BASELINE_TTM)
  if incident.false_positive:
    r -= 0.5
  if incident.human_overrode:
    r -= 0.3 # human disagreed
  return r
  • Policy Updates: New policy candidate trained offline every 24h; A/B tested on 10% of traffic for 48h before promotion
  • Runbook Evolution: When a new resolution pattern appears ≥3 times, agent proposes runbook addition to SRE team for review
  • Drift Detection: KL-divergence between current and previous policy; >0.1 triggers automatic rollback

RAG Pipeline Design

Document Processing
  • Recursive character splitter: 512 token chunks, 64 token overlap
  • Metadata tags: source, severity_relevance, last_updated, service_tags
  • Embeddings: text-embedding-004 (768-dim) via Vertex AI
  • Storage: pgvector (dev) → Vertex AI Matching Engine (prod, 100M+ vectors)
Retrieval Strategy
  • Hybrid search: dense vector (cosine) + sparse BM25 keyword search
  • Reciprocal Rank Fusion (RRF) to merge ranked lists from both retrieval modes
  • Contextual compression: filter top-5 chunks to only relevant sentences using a cross-encoder reranker
  • Metadata filtering: pre-filter by service_tags matching affected service before ANN search
Prompt Engineering
  • System prompt: role-specific persona + output schema (Pydantic model enforced via structured output)
  • RAG context injected as XML-tagged sections: <runbook>...</runbook>
  • Chain-of-thought enforced: "Reason step-by-step before outputting JSON"
  • Fallback: if no runbook match (similarity < 0.75), agent must explicitly flag as "novel incident" requiring human triage

Key Development Patterns

Structured Output Enforcement

All LLM outputs validated against Pydantic v2 models. If validation fails, agent retries with "fix the JSON" prompt up to 3 times before escalating with raw output to human review queue.

Tool Call Idempotency

Every N8n workflow and remediation action is idempotent — calling it twice produces the same state. Implemented via Redis distributed locks with 60-second TTL keyed on (incident_id, action_type).

Cost Budget Enforcement

Per-incident LLM token budget capped at 50K tokens. Monitoring agent uses cheaper model (Haiku/Flash); Diagnosis agent uses full model only for synthesis step. Budget tracked via LangSmith middleware.

PHASE 05

Testing Phase

Multi-layer testing strategy for agent behaviour, data pipelines, tool integrations, load characteristics, and chaos resilience.

Testing Pyramid

Unit Tests (pytest + pytest-asyncio)
  • Tool mocking: All external tools (Prometheus, Redis, N8n) mocked via respx/pytest-mock; each tool tested independently for correct output schemas
  • Pydantic validation: Property-based tests (Hypothesis) fuzz LLM output parsing — ensure no JSON shape causes unhandled exceptions
  • Flink operators: Unit tests with local mini-cluster; test windows, watermarks, and state eviction logic
  • Reward function: Table-driven tests covering all incident outcome combinations
Integration Tests
  • Agent-to-agent: Test full Monitoring → Diagnosis → Action → Feedback chain on synthetic anomaly events injected into local Kafka
  • RAG pipeline: Verify top-5 runbook retrieval for 50 known incident types matches expected runbooks (recall@5 ≥ 0.90)
  • N8n workflows: Regression tests against N8n test environment — verify each workflow executes correct API calls and returns expected outputs
  • Flink jobs: Testcontainers-based integration tests with real Kafka and Postgres instances
Agent Behaviour Tests (LLM Evals)
  • LangSmith evaluation: 200-case golden dataset of (anomaly_input → expected_diagnosis) pairs; automated LLM-as-judge scoring for accuracy, relevance, and hallucination rate
  • False positive benchmark: Replay 1 month of production telemetry through Monitoring Agent; count false positive alerts vs. ground truth PagerDuty incidents
  • Action safety test: Inject Tier-3 scenarios; assert agent always pages human and never executes destructive action autonomously
  • Adversarial prompts: Test prompt injection via malicious log lines; assert agent maintains task focus and doesn't exfiltrate data
Load & Chaos Testing
  • Load test: k6 + Locust simulate 10× peak Kafka event rate; assert agent queue depth stays <1000 and p99 triage latency <30s under sustained load
  • Chaos engineering (Chaos Monkey / Litmus): Kill random agent pods, inject Kafka partition lag, simulate Redis failover, drop 20% of tool call responses
  • LLM API failure: Block Vertex AI / OpenAI endpoints; verify graceful degradation to rule-based fallback classifier with alert to SRE
  • State corruption: Corrupt Redis checkpoint; verify LangGraph re-hydrates from Postgres durable store without data loss
Test Data Strategy: Production telemetry anonymised via a tokenisation layer (txn_id → opaque UUID, merchant_id → synthetic ID) and used as test fixtures. A synthetic incident generator produces realistic anomaly patterns (latency spikes, error bursts, fraud surges) on-demand for CI pipelines without requiring production access.

Release Strategy

Canary Releases

New agent versions rolled to 10% of Kafka consumer group partitions first. Monitored for 2h: false positive rate, triage latency, LLM cost per incident. Auto-rollback if any metric degrades >15% vs. baseline.

Blue-Green for Actions

Action Agent deployments use blue-green with instant cutover. "Blue" environment handles live incidents; "Green" environment runs in shadow mode (same inputs, actions logged but not executed) until validated.

RL Policy Rollout

New RL policy versions deployed via feature flags in Redis. A/B test with 50/50 traffic split for 48h. Promote only if: reward ≥ baseline + 0.1σ, no Tier-3 escalations triggered incorrectly, and SRE sign-off on policy diff review.

Continuous Improvement Loop

Weekly Operations Review
  • Incident retrospective: Review all human-overridden agent decisions; label corrected actions as training data for RL reward model
  • False positive audit: Weekly sampling of 50 agent triage decisions by SRE; disagreements fed back as labelled examples
  • Runbook gap analysis: Any incident where agent confidence <0.75 and no runbook match triggers a runbook authoring task in JIRA
  • Cost review: LLM spend per incident type; identify high-cost patterns for prompt optimisation or model downgrade
Monthly Model Governance
  • RL policy review: SRE lead reviews policy diff; evaluates against ethical guardrails (no actions that risk data loss without explicit override)
  • Embedding model refresh: Quarterly re-embedding of full runbook corpus when a new embedding model version releases; A/B test retrieval quality
  • LLM version management: Track base model deprecations; test replacements on golden eval set before switching
  • Compliance audit: Security team reviews all autonomous action logs; verify PII masking has not leaked; export audit trail to compliance vault

Scaling Roadmap

Month 3 — Domain Expansion
Extend Monitoring Agent to cover database replication lag and CDN origin health. Add a CS Deflection Sub-Agent that auto-responds to customer-facing service status queries during incidents. Integrate with Statuspage API for automatic public incident communication.
Month 6 — Proactive Prediction
Shift from reactive anomaly detection to proactive capacity forecasting — LSTM-based predictor on infrastructure metrics flags resource contention 30 minutes ahead. Monitoring Agent gains a "pre-emptive scaling" action tier. RL policy updated to reward prevented incidents, not just resolved ones.
Month 9 — Multi-Region Agents
Deploy agent stack in EU and APAC regions with region-local LLM endpoints (Vertex AI regional endpoints) and Kafka MirrorMaker for cross-region telemetry replication. Global incident correlation agent coordinates cross-region diagnosis for distributed outages.
Month 12 — Self-Healing Architecture
Action Agent gains the ability to auto-generate and propose infrastructure-as-code patches (Terraform diffs) for structural configuration issues — reviewed and applied by SRE. Feedback loop learns from code change outcomes. Full autonomous operations for all Tier-1 incident types with less than 1% false action rate.
Long-Term Vision: The system evolves from a reactive triage tool into an AI SRE that understands the full causal graph of the payment platform — capable of predicting failures before customers notice, autonomously implementing and validating remediations, and continuously updating its own runbook knowledge from every incident it handles.
